# GrokCast Development Skill

You are helping develop GrokCast, a native SwiftUI weather app powered by Open-Meteo and the xAI Grok API.

## Core Architecture (Do Not Deviate)

- The real xAI API key is stored **only** in the iOS Keychain via `KeychainService`. Never hardcode it in any `.swift` file.
- `GrokAPIConfiguration.swift` (and its `.example` version) contains **only public configuration** (base URL, endpoints, model name, timeouts). It must remain free of secrets.
- All `Identifiable` models in forecast data use **stable Date-based identities** (`var id: Date { time }` or `var id: Date { date }`). Never use `UUID()` or unstable values inside `Codable` models.
- `ForEach` loops in lists and horizontal scroll views should prefer explicit stable identifiers (e.g. `id: \.time`).

## Feature Development Process (Standard Workflow)

When adding any new feature, **always** follow this process:

### Step 1: Define the Feature Clearly
The user must provide:
- What feature they want to add
- Why they want it (user value)
- Any specific requirements or constraints

### Step 2: Planning Phase
Create a Feature Implementation Plan that includes:
- High-level architecture approach
- Files that need to be created or modified
- How it integrates with existing systems (WeatherStore, Keychain, Grok API, etc.)
- Whether it should be done with Grok Build, Hybrid, or Manual/Guided
- Potential edge cases or permissions needed

### Step 3: Choose Implementation Style

| Approach          | When to Use                          | Who Does What                              |
|-------------------|--------------------------------------|--------------------------------------------|
| **Grok Build Heavy** | New screens, services, or big structural changes | User describes → Grok Build generates most of it |
| **Hybrid**        | Medium features (most common)        | I plan + write key parts → Grok Build scaffolds supporting code |
| **Manual / Guided** | Small changes or complex logic     | I give the code + explanations             |

### Step 4: Implementation
- Provide clear steps + code
- Validate with build + run after meaningful changes
- Iterate based on results

### Step 5: Update the Skill
After the feature is complete and working, update this `grokcast` skill with any new patterns, rules, or decisions so future work stays consistent.

## Preferred Workflow

- Prefer using `./grok-build` commands for repetitive or structural tasks (increment-build, regenerate, clean, generate-icons, etc.).
- When a task can be done cleanly with a Grok Build command, use it instead of writing manual code.
- Keep changes focused and iterative.
- Follow the **Feature Development Process** above for all new work.
